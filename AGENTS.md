# AGENTS.md — Text Diffusion Lab

## Project identity

This repository is for **Text Diffusion Lab**, an Open Day AI demo for the IT marquee.

The demo is part of the wider **Open Day Demos** project. Its public-facing title should be:

> **Beyond Next-Word Prediction: Text Diffusion Lab**
>
> **A glimpse at one possible future for generative AI.**

The demo should show visitors that generative AI does not have to be presented only as left-to-right next-token generation. The MVP should show text improving through repeated refinement: noisy text, rough draft, clearer draft, styled draft, final answer.

## Build priority

Build in this order:

1. **Phase A — From noise to meaning**
   - Build a visually compelling staged refinement loop.
   - Use scripted traces first.
   - No live model is required for the first working version.

2. **Phase B — Visitor-controlled refinement**
   - Add curated prompt cards.
   - Add simple controls: style, creativity, length, constraint, speed/steps.
   - Prefer buttons and sliders over free text.

3. **Phase C — Compare with today’s LLMs**
   - Add a split-screen comparison between left-to-right generation and staged refinement.
   - Only start this after A+B are clear, reliable, and visually strong.

Do **not** start with a true text-diffusion backend. Treat a real diffusion language model as optional research/stretch mode only.

## Public explanation

Use this framing:

> “Most chatbots generate text one piece at a time. Text diffusion is a research direction that explores a different approach: start with rough, noisy, or incomplete text, then repeatedly refine it. This demo is simplified, but it shows why future GenAI might become more controllable, editable, or creative.”

Use these terms:

- “simplified demo”
- “diffusion-inspired”
- “staged refinement”
- “one possible future direction”
- “not necessarily how all future AI will work”

Avoid these claims:

- “This is definitely the future of AI.”
- “This shows the AI thinking.”
- “This reveals private model reasoning.”
- “Diffusion is better than LLMs.”
- “This is a clone of Gemini Diffusion.”
- “This cannot fail.”

## UX principles

The demo must be understandable within **30–45 seconds**.

The screen should be readable from **2–3 metres away**.

Every run should clearly show that earlier words can change. This is the key distinction from a left-to-right token stream.

Prefer a big-screen layout like:

```text
Prompt: A robot joins university orientation.
Style: Funny     Creativity: 65%     Steps: 6

Step 0 — Noise
robot / lost / campus / ??? / friend / map / toaster

Step 1 — Rough idea
A robot was lost at university and needed a map.

Step 2 — Clearer
A robot got lost during orientation and asked students for help.

Step 3 — Styled
A robot joined orientation, downloaded the campus map, and still followed a duck to the wrong lecture.

What this shows:
Text can be generated by repeated refinement, not only by adding one word at a time.
```

## Inputs

Preferred input modes, safest first:

1. Prompt cards.
2. Style buttons.
3. Sliders.
4. Staff-entered prompt.
5. Short constrained visitor text.

Do not enable open free text by default for Open Day mode.

Suggested prompt cards:

- “A robot joins university orientation.”
- “A reef scientist discovers a strange signal.”
- “A student builds an app overnight.”
- “A tiny AI helps in a practical class.”

Suggested styles:

- Clear
- Funny
- Sci-fi
- Campus
- Scientific
- Poetic

Suggested constraints:

- Include “reef”.
- Include “robot”.
- Include “university”.
- Keep it under 12 words.
- Make it rhyme.

## Required modes

Implement these modes early:

| Mode | Purpose | Required for MVP |
|---|---|---:|
| Scripted replay | Bulletproof fallback and demo loop | Yes |
| Template refinement | Reliable live-feeling generation without model dependency | Yes |
| Model-assisted refinement | Optional local SLM rewrite stages | Later |
| Split-screen comparison | Phase C teaching layer | Later |

Fallback mode must work offline and must not depend on a model call.

## Ports and local services

Use the Open Day reserved ports for Text Diffusion Lab:

```env
DEMO_NAME=text-diffusion-lab
DEMO_MODE=development

FRONTEND_HOST=127.0.0.1
FRONTEND_PORT=3300

BACKEND_HOST=127.0.0.1
BACKEND_PORT=8300
BACKEND_URL=http://127.0.0.1:8300

MODEL_ADAPTER_URL=http://127.0.0.1:8600
REPLAY_SERVICE_URL=http://127.0.0.1:8700
```

Do not silently fall back to random ports in Open Day mode.

Avoid unplanned use of `3000`, `5000`, `5173`, `8000`, or `8080` for demo services.

## Data and privacy rules

Default position:

- Do not collect names, emails, phone numbers, or identifiable visitor data.
- Do not require login.
- Keep logs technical and non-identifying.
- Clear visitor-visible text on reset.
- Avoid open free text unless staff-supervised.
- Do not store visitor prompts longer than needed for the current session unless explicitly documented.

## Accessibility and booth-readiness

The UI should use:

- large text;
- high contrast;
- obvious stage labels;
- visible reset and replay buttons;
- reduced-motion option if animation becomes distracting;
- keyboard-accessible controls where practical.

The public screen must always show:

- current mode: live, scripted, or replay;
- what the demo shows;
- a clean reset state;
- no private or inappropriate visitor input.

## Development workflow for coding agents

When asked to implement a change:

1. Inspect the existing files first.
2. Propose a short plan before editing code.
3. Make small incremental changes.
4. Prefer scripted/fallback behaviour before model-dependent behaviour.
5. Add or update tests for changed logic.
6. Run the relevant checks.
7. Summarise changed files and any checks that could not be run.

Do not add unnecessary dependencies.

Do not introduce hosted services unless explicitly requested.

Do not remove fallback/replay behaviour.

Do not add storage of visitor data without explicit approval and documentation.

## Acceptance criteria

The MVP is acceptable when:

- at least three curated prompts produce clean staged outputs;
- the refinement process is visually obvious;
- staff can explain the point in under 45 seconds;
- reset clears the current prompt and stages;
- scripted replay works offline;
- the app runs on the reserved Text Diffusion ports;
- simulated mode looks intentional, not fake or broken;
- public wording remains modest and accurate.

## Phase C acceptance criteria

The comparison view is acceptable when:

- it clearly contrasts left-to-right token generation with staged whole-text refinement;
- it does not imply that one approach is always better;
- it explains that diffusion language models are a research direction, not a guaranteed replacement for LLMs;
- it can be disabled or hidden if it confuses visitors.
